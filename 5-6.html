📦 02 · Lecture 5

UML Class Diagrams

دي الـblueprint الأساسية للنظام. لما تتقن المفاهيم دي، تقدر تصمم أي نظام Object-Oriented على الورق قبل كتابة code.

1
What is UML?
▾

Unified Modelling Language (UML) هي لغة رسومية قياسية لنمذجة أنظمة Object-Oriented. اعتبرها grammar مشتركة: أي developer يرسم UML diagram، أي حد تاني يقدر يفهمه بنفس المعنى.

Official definition: UML هي standard graphical language لنمذجة Object-Oriented software. اتطورت بواسطة Rumbaugh, Booch, Jacobson وتم توحيدها بواسطة OMG سنة 1997.

UML Diagram Types:

Diagram View Purpose
Class Diagrams Static Describe classes and their relationships
Sequence Diagrams Dynamic Show how objects interact in time
Communication Diagrams Dynamic Show object interactions (different layout)
State Diagrams Dynamic Show internal behaviour of a class
Activity Diagrams Dynamic Show workflow / process flow
Component Diagrams Physical Logical arrangement of components
Deployment Diagrams Physical Physical arrangement of hardware/software
Use Case Diagrams Functional System behaviour from user perspective
2
Classes, Attributes & Operations
▾

الـClass بتمثل بصندوق فيه 3 أجزاء: الاسم، attributes، وoperations. وهي الـtemplate اللي بنبني منه objects.

BankAccount
- accountNumber: String
- balance: double
- owner: String
+ deposit(amount: double): void
+ withdraw(amount: double): boolean
+ getBalance(): double
Attributes — بيانات بسيطة بتتخزن داخل الـclass والinstances بتاعتها (مثال balance: double)
Operations — functions بتنّفذها الـclass. الصيغة الكاملة للـsignature هي:
operationName(paramName: paramType…): returnType
Visibility Modifiers: + public  ·  - private  ·  # protected  ·  ~ package
⚠️

امتى نعرض التفاصيل؟ في مستوى design بنعرض attributes وmethods. في domain analysis ممكن نكتفي باسم الـclass. في implementation بنعرض كل التفاصيل بما فيها getters/setters.

3
Associations & Multiplicity
▾

الـassociation بتوضح علاقة كلاسين ببعض. وقت التشغيل، instance من كلاس ممكن يرتبط بـinstance من كلاس تاني. رموز multiplicity بتوضح عدد الـinstances في العلاقة.

Notation Meaning Example
1 Exactly one An employee works for exactly 1 company
0..1 Zero or one A person may or may not have a spouse
* or 0..* Zero or more A company can have any number of employees
1..* One or more A project must have at least one developer
2..5 Between 2 and 5 A team has 2 to 5 members

Common Association Types with examples from the slides:

Many-to-One (Many employees work for one company)

Employee
* ———— worksFor ———— 1
Company
Employee can work for 0 or 1 company? No — reading the slide: "It is not possible to be an employee unless you work for a company" → Employee must have exactly 1 company.

One-to-One (A company has exactly one Board of Directors)

Company
——— 1 ———————— 1 ———
Board of Directors
⚠️

تجنب 1-to-1 associations غير الضرورية! لو حاجتين دايمًا one-to-one ومش بينفع واحدة تعيش بدون الثانية، الأفضل غالبًا دمجهم في class واحدة بattributes أكتر.

Association Classes: Sometimes an attribute belongs to the relationship itself, not to either class. In this case, attach an Association Class to the association line. Example: the grade a student gets in a course belongs to the enrollment, not to Student or Course alone.

Reflexive Associations: A class can have an association with itself! Example: A Course has prerequisite Courses (the same Course class links to itself).

Directionality: Associations are bi-directional by default. Add an arrowhead to make it one-way (navigable in only one direction).

4
Generalization (Inheritance Hierarchies)
▾

Generalization هو مصطلح UML للـ inheritance. الـsuperclass بيتخصص إلى subclass أو أكتر، والسهم دايمًا بيشير للـsuperclass (hollow triangle).

Generalization Set: A labeled group of generalizations with a common superclass. The label (called the discriminator) describes the criteria used in the specialization.
Employee
- name: String
- salary: double
Manager
- budget: double
Developer
- language: String
⚠️

قاعدة مهمة: الـinstance المفروض ما تحتاجش تغير class. لو الشخص ينفع يكون Manager و Developer معًا (أو يبدّل roles)، غالبًا generalization مش الاختيار الصح واستخدم association أو attribute بدلها. دي trap مشهورة في الامتحان.

Handling Multiple Discriminators: Two strategies:

  • Higher-level generalization: Create an intermediate superclass
  • Multiple inheritance: A class inherits from multiple parents (e.g., a FlyingFish inherits from both Bird and Fish)

In Object Diagrams: Generalizations do NOT appear — they describe class-level relationships. An instance of Manager IS ALSO an instance of Employee automatically.

5
Object Diagrams (Instance Diagrams)
▾

الـObject Diagram (أو Instance Diagram) بيوضح instances حقيقية من classes في لحظة زمنية معينة. الـlink هو instance من association، زي ما object هو instance من class.

Associations describe relationships between instances at runtime → they APPEAR in object diagrams.
Generalizations describe relationships between classes → they do NOT appear in object diagrams.
john: Employee
name = "John Smith"
salary = 60000
acme: Company
name = "Acme Corp"

Note: Object names are underlined in UML. Format: objectName: ClassName

6
Aggregation & Composition (Part-Whole)
▾

Aggregation وComposition نوعين خاصين من associations بيمثلوا علاقة part-whole، يعني class مكوّنة من classes أخرى.

🔷 Aggregation (Open Diamond ◇)
The whole and parts have an independent lifecycle. Parts can exist without the whole.
Example: Car ◇— Wheel (wheels can be sold separately)
🔶 Composition (Filled Diamond ◆)
Strong aggregation — if the aggregate is destroyed, the parts are destroyed too. Parts cannot exist independently.
Example: Building ◆— Room (no room without a building)
⚠️

The Diamond Side = The Whole (Aggregate). الدايموند دايمًا على طرف الـwhole مش الـpart، ودي نقطة كتير من الطلبة بيغلطوا فيها.

When to use aggregation? Use it when:

  • You can say "the parts ARE PART OF the aggregate"
  • OR "the aggregate IS COMPOSED OF the parts"
  • AND "when something owns the aggregate, they also own the parts"

Propagation: A mechanism where an operation on an aggregate is implemented by having the aggregate perform that operation on its parts. For example, calling calculateTotalWeight() on a Car propagates to summing the weights of all its parts.

🔗 Connections: Relationship Summary
▾
Relationship Symbol Key Question Example
Association Solid line "Are they related?" Employee works for Company
Generalization Hollow arrow → superclass "Is it a type of?" Manager IS-A Employee
Aggregation Open diamond ◇ on whole "Is it part of (independently)?" Wheel part of Car
Composition Filled diamond ◆ on whole "Is it part of (dependently)?" Room part of Building
Reflexive Assoc. Line loops back to same class "Does it relate to itself?" Course has prerequisite Courses
Directed Assoc. Arrow on one end "Is navigation one-way?" Client → Account (client knows about account but not vice versa)
💡

The body of a person = Composition (body parts cannot live without the person).
Parts of a car = Aggregation (wheels, engine can be sold separately). These two examples appear directly in the slides and in exercises!

⏱ 03 · Lecture 6

Sequence Diagrams

الفيلم الديناميكي للنظام: إزاي الـobjects بتتبادل messages خطوة بخطوة عشان تنفذ use case كاملة.

1
What is Modeling? Why Does It Matter?
▾
Modeling يعني بناء abstraction للواقع. الـabstraction بتبسط الصورة: بنشيل التفاصيل غير المهمة ونحتفظ باللي يخدم هدف النموذج. المهم بيتحدد حسب purpose.

فكر فيها زي street map: الخريطة مش بتعرض كل شجرة ولا ارتفاع كل مبنى، لكنها بتعرض ما يكفي للملاحة. نفس الفكرة في software models: بنعرض فقط ما يخدم قرار التصميم.

Why model software?

  • Software is getting massively complex (Windows 10 → 50+ million lines of code)
  • No single developer can manage that much code alone
  • Code written by others is hard to understand without diagrams
  • Models are simpler representations of complex systems
Application Domain — The environment in which the system operates (Requirements Analysis phase)

Solution Domain — The available technologies to build the system (System Design phase)
2
Interaction Diagrams (Section 8.1)
▾
Interaction diagrams are used to model the dynamic aspects of a software system. They help visualize how the system runs. An interaction diagram is often built from a use case (user story) AND a class diagram.

The objective is to show how a set of objects accomplish the required interactions with an actor. There are two kinds:

  • Sequence Diagrams — organize messages along a time axis (top to bottom)
  • Communication Diagrams — same info, different layout (graph-based)
⚠️

Pre-requisite: You should develop a class diagram AND a use case model BEFORE creating an interaction diagram. Interaction diagrams show how instances of those classes collaborate.

3
Sequence Diagram Elements
▾

A sequence diagram shows the sequence of messages exchanged by a set of objects performing a certain task.

Element Appearance Meaning
Participants Boxes at the top Instances of classes or actors (underlined names)
Lifeline Vertical dashed line below each participant Represents time passing for that object
Activation Box Thin rectangle on lifeline Period when an object is active (executing)
Message Horizontal arrow between lifelines Method call — labelled with name, args, return value
Return Message Dashed arrow Return value from a method call
Frame Outer rectangle with label Groups the entire diagram; labeled with diagram name
Key Layout Rules:
• Objects arranged horizontally
• Actor that initiates interaction on the LEFT
• Time flows downward (vertical axis = time)
• Each participant has a lifeline (vertical dashed line)

4
Lifeline Naming Conventions
▾

How you name a lifeline tells the reader important information about the object:

Notation Meaning
seoclecturer An object named seoclecturer (no class specified)
seoclecturer:Lecturer An object named seoclecturer of class Lecturer
:Lecturer An anonymous object of class Lecturer
lecturer[i]:Lecturer The object lecturer selected by index value i
⚠️

Lifeline names are always underlined in UML — this is how you distinguish instances (object names) from classes.

5
Element Creation & Destruction
▾
Element Creation: When an element is created during an interaction, the communication that creates the element is shown with its arrowhead pointing to the element (which starts in the middle of the diagram, not at the top).
Element Destruction: When an element is destroyed during an interaction, the communication that destroys the element points to the element's lifeline where the destruction is marked with a large X symbol.
Replicated Messages: When the same message is sent multiple times (e.g., to multiple objects), this can be shown with loop notation or replicated arrows.
6
Why Sequence Diagrams Instead of Just Coding?
▾

سؤال مهم من السلايدز: ليه نرسم diagrams طالما نقدر نكتب code مباشرة؟

  • A good sequence diagram is above code level — it shows WHAT happens without committing to HOW it's implemented
  • Language-independent — can be implemented in Java, Python, C++, or any language
  • Non-coders can read them — clients and managers can review the logic
  • Visual representation — easier to spot problems in flow than in code
Difficulties in Dynamic Modelling: In large systems, there are a very large number of possible paths. It's hard to allocate behaviour to the right classes. Solution: Work iteratively — develop class diagrams, use cases, interaction diagrams; then go back and verify consistency.

🧩 Deep Understanding — Common Mistakes
❌ Mistake #1: Wrong Diamond Side

The diamond (◆ or ◇) always goes on the WHOLE side, not the part side. Building ◆—Room, NOT Room ◆—Building.

❌ Mistake #2: Generalization vs. Association

Use generalization (arrow) for IS-A. Use association (line) for HAS-A or USES-A. An employee IS-A person. An employee HAS-A contract.

❌ Mistake #3: Reading Multiplicity Backwards

The multiplicity is read from the OTHER class's perspective. In "Employee (*) ——— (1) Company", a Company can have many Employees, not the other way.

❌ Mistake #4: Changing Classes

An instance should NEVER need to change class. If a person could be both student and employee simultaneously, don't use generalization — use role attributes or associations.

❌ Mistake #5: Generalizations in Object Diagrams

Generalization does NOT appear in object/instance diagrams. Only associations appear as links between specific objects.

❌ Mistake #6: Agg vs Comp for Bodies

Person's body parts = COMPOSITION (cannot live without person). Car's parts = AGGREGATION (can be sold separately). The key question: can parts exist independently?
